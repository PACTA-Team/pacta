# Contributing to PACTA

Thank you for your interest in contributing to PACTA! This document provides guidelines and information for contributors.

## Getting Started

### Prerequisites

- **Go 1.25+** (with CGO enabled for local AI features)
- **Node.js 22+** and npm
- **CMake** and C compiler (for llama.cpp CGo bindings)
- **git**

See [INSTALLATION-LINUX.md](docs/INSTALLATION-LINUX.md) or [INSTALLATION-WINDOWS.md](docs/INSTALLATION-WINDOWS.md) for setup details.

### Development Setup

```bash
# 1. Clone repository
git clone https://github.com/PACTA-Team/pacta.git
cd pacta

# 2. Install frontend dependencies
cd pacta_appweb
npm ci

# 3. Build frontend
npm run build
cd ..

# 4. Run Go server (auto-reload not supported — rebuild frontend on changes)
go run ./cmd/pacta
```

The app runs at `http://127.0.0.1:3000`.

## Database Queries (sqlc)

PACTA uses **sqlc v2** for type-safe database access. All SQL queries live in `.sql` files and generate Go code.

### Structure

```
internal/db/
├── migrations/          # goose migrations (versioned schema changes)
├── models.go            # Go structs matching table rows
├── queries/             # SQL files by domain (SOURCE OF TRUTH)
│   ├── system_settings.sql
│   ├── users.sql
│   ├── contracts.sql
│   └── ...
├── queries_gen.go       # GENERATED — type-safe methods (DO NOT EDIT)
├── sqlc.yaml            # sqlc configuration
└── db.go                # Open() and Migrate() helpers
```

### Adding a New Query

1. **Create or edit** a `.sql` file in `internal/db/queries/`

   Example (`system_settings.sql`):
   ```sql
   -- name: GetSettingValue :one
   SELECT value FROM system_settings
   WHERE key = $1 AND deleted_at IS NULL
   LIMIT 1;
   ```

   **Naming conventions** (`:one`, `:many`, `:exec`, `:execrows`):
   - `:one` — returns single row (use `RETURNING *` for inserts/updates)
   - `:many` — returns multiple rows
   - `:exec` — no return data (INSERT/UPDATE/DELETE without RETURNING)
   - `:execrows` — returns `sql.Result` (rows affected)

2. **Generate code**:
   ```bash
   cd internal/db
   sqlc generate
   ```

   This updates `queries_gen.go` with a new method:
   ```go
   func (q *Queries) GetSettingValue(ctx context.Context, key string) (string, error)
   ```

3. **Use it in a handler**:
   ```go
   value, err := h.queries.GetSettingValue(r.Context(), "setting_key")
   ```

4. **Commit** both:
   ```bash
   git add internal/db/queries/system_settings.sql internal/db/queries_gen.go
   git commit -m "feat: add GetSettingValue query"
   ```

### Modifying Existing Queries

1. Edit the corresponding `.sql` file in `internal/db/queries/`
2. Run `sqlc generate`
3. Update any handlers/tests that call the query (signature may have changed)
4. Run tests: `go test ./...`
5. Commit both the `.sql` file and `queries_gen.go`

**Important**: Always verify the generated method signature in `queries_gen.go` before updating callers.

### Soft-Delete Pattern

All tables with `deleted_at TIMESTAMP` **must** include `AND deleted_at IS NULL` in SELECT queries:

```sql
SELECT * FROM users
WHERE id = $1 AND deleted_at IS NULL
LIMIT 1;
```

**Exceptions**: Admin endpoints that need to see deleted records (rare).

### Transactions

Use the transaction-aware interface generated by sqlc:

```go
tx, err := h.queries.BeginTx(ctx)
if err != nil { /* handle */ }
defer tx.Rollback()

// Use tx.* methods (same signatures as queries)
user, err := tx.GetUserByID(ctx, id)
if err != nil { /* handle */ }

if err := tx.CreateAuditLog(ctx, CreateAuditLogParams{...}); err != nil {
    return err
}

if err := tx.Commit(); err != nil {
    return err
}
```

### Testing with sqlc

`db.Queries` is an interface → easy to mock:

```go
type mockQueries struct{}

func (m *mockQueries) GetUserByID(ctx context.Context, id int) (User, error) {
    return User{ID: id, Name: "Test"}, nil
}

// In test:
handler := NewHandler(&sql.DB{}, &mockQueries{}, ...)
```

## Database Migrations (goose)

Migrations live in `internal/db/migrations/` and run automatically on startup.

### Adding a Migration

1. Create a new file: `internal/db/migrations/NNN_description.sql`
   - `NNN` = sequential number (e.g., `001_initial.sql`, `002_add_users_table.sql`)
   - Use snake_case for description

2. Write **up** and **down** statements:

   ```sql
   -- +goose Up
   CREATE TABLE users (
       id INTEGER PRIMARY KEY AUTOINCREMENT,
       email TEXT NOT NULL UNIQUE,
       created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
   );

   -- +goose Down
   DROP TABLE users;
   ```

3. No manual execution needed — `db.Migrate()` runs all pending migrations on startup.

4. Commit migration file:
   ```bash
   git add internal/db/migrations/042_add_ai_settings.sql
   git commit -m "feat(db): add ai_settings table"
   ```

**Note**: Never modify an already-applied migration in production. Create a new migration instead.

## Frontend Development

PACTA's frontend lives in `pacta_appweb/`:

- **Framework**: React 19 + TypeScript
- **Build tool**: Vite
- **Styling**: Tailwind CSS v4 + shadcn/ui components
- **Animations**: Framer Motion
- **Routing**: File-based (app directory)

### Running Dev Server

```bash
cd pacta_appweb
npm run dev
```

The dev server runs on `http://localhost:5173` (or configured port) with hot reload.

### Adding a Component (shadcn/ui)

```bash
cd pacta_appweb
npx shadcn@latest add button
# or: add dialog, card, table, etc.
```

Components are added to `src/components/ui/`.

## Commit Guidelines

Follow [Conventional Commits](https://www.conventionalcommits.org/):

```
type(scope): description

feat(db): add rate_limit column to users table
fix(handlers): correct password reset token expiry
refactor(ai): simplify MiniRAG context building
docs(README): add sqlc migration section
chore(deps): update golangci-lint
```

**Types**: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `build`, `ci`, `revert`

**Scope** (optional): `db`, `handlers`, `ai`, `email`, `frontend`, `models`, etc.

## Code Style

### Go

- Use `gofmt` (enforced by CI)
- Run `go vet ./...` before committing
- Follow [Effective Go](https://golang.org/doc/effective_go)
- Error messages: start with lowercase, no punctuation: `err := fmt.Errorf("connection failed")`

### TypeScript / React

- Use explicit types for function props and state
- Components: function declarations or arrow functions (consistent)
- Imports: external first, then internal, sorted alphabetically
- Tailwind classes order: layout → typography → spacing → colors → effects

## Pull Request Process

1. **Fork** the repository (if external contributor) or create a **feature branch**:
   ```bash
   git checkout -b feat/short-description
   ```

2. **Make changes** following guidelines above.

3. **Run tests locally**:
   ```bash
   go test ./...
   cd pacta_appweb && npm run build
   ```

4. **Commit** with clear, descriptive messages (Conventional Commits).

5. **Push** and open a **Pull Request** against `main` branch.

6. **PR Template** (fill all sections):
   ```markdown
   ## Summary
   What does this PR change?

   ## Motivation
   Why is this change needed? Problem solved?

   ## Testing
   How was this tested? Steps to reproduce verification?

   ## Screenshots (if UI change)
   Before / After

   ## Checklist
   - [ ] Tests pass (`go test ./...`)
   - [ ] Frontend builds (`npm run build`)
   - [ ] No new raw SQL queries (use sqlc)
   - [ ] Commit follows Conventional Commits
   - [ ] Documentation updated (if needed)
   ```

7. **Code Review**: Maintainer will review. Address feedback, push fixes as new commits.

8. **Merge**: PR is squash-merged into `main` by project maintainer.

## Performance & Security

### Performance

- Use indexes for frequent queries (check `EXPLAIN QUERY PLAN`)
- Avoid N+1 queries in handlers (use joins or batch queries)
- Soft deletes require `deleted_at IS NULL` in every SELECT — add index if needed: `CREATE INDEX idx_table_deleted_at ON table(deleted_at) WHERE deleted_at IS NULL;`

### Security

- Never log passwords, API keys, or sensitive PII
- Validate all user input ( handlers already do this with `validator` tags )
- Use prepared statements (sqlc does this automatically)
- Session cookies are `HttpOnly` + `Secure` in production
- CSRF protection on state-changing endpoints

See [SECURITY.md](SECURITY.md) for vulnerability disclosure.
