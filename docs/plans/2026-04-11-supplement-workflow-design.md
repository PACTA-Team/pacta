# Supplement Workflow — Full Stack Design

**Date:** 2026-04-11
**Status:** Approved
**Scope:** Backend API endpoints + Frontend migration from localStorage to API

---

## Problem

Supplements have a database schema (`006_supplements.sql`) but no backend API endpoints. The frontend `SupplementsPage.tsx` uses localStorage, which means:
- Data doesn't persist across browsers or machines
- No audit trail for supplement changes
- No integration with the backend contract data
- No status workflow enforcement

---

## Design Decisions

### Dual Identifier Pattern (like contracts)

| Field | Purpose | Format | Generated By |
|-------|---------|--------|--------------|
| `supplement_number` | Legal number from user's legal team | Free text (e.g., "SUP-2026-001", "Adenda 3") | User (manual entry) |
| `internal_id` | System tracking ID | `SPL-YYYY-NNNN` (e.g., `SPL-2026-0001`) | System (auto-generated) |

**Rationale:** Legal teams use their own numbering schemes. PACTA needs a reliable internal ID for audit trails, API references, and database integrity.

### Status Transition Rules

| From | To | Allowed | Use Case |
|------|----|---------|----------|
| draft | approved | Yes | Approve supplement |
| approved | draft | Yes | Return for corrections |
| approved | active | Yes | Activate supplement |
| active | any | No | Terminal state |
| draft | active | No | Must go through approval |

**Rationale:** Supplements need a review cycle. Approved supplements can return to draft if errors are found before activation. Once active, the supplement is legally binding and cannot be reverted.

### API Structure

Both full CRUD (PUT) and dedicated status endpoint (PATCH):
- **PUT** for editing supplement data (description, date, signers, modifications)
- **PATCH /status** as a workflow shortcut for approval flow
- Separation allows cleaner audit logging and permission checks

## Backend Design

### Migration: `012_supplements_internal_id.sql`

```sql
-- Add internal_id column
ALTER TABLE supplements ADD COLUMN internal_id TEXT;

-- Backfill existing records
UPDATE supplements SET internal_id = 'SPL-' || strftime('%Y', created_at) || '-' ||
    printf('%04d', (
        SELECT COUNT(*) FROM supplements s2
        WHERE s2.id <= supplements.id
        AND strftime('%Y', s2.created_at) = strftime('%Y', supplements.created_at)
    ))
WHERE internal_id IS NULL;

-- Make NOT NULL and unique
CREATE UNIQUE INDEX idx_supplements_internal_id ON supplements(internal_id);
CREATE INDEX idx_supplements_internal_id_year ON supplements(SUBSTR(internal_id, 1, 8));
```

### Internal ID Generation

```go
func (h *Handler) generateSupplementInternalID() (string, error) {
    year := time.Now().Year()
    var maxNum sql.NullInt64
    err := h.DB.QueryRow(`
        SELECT MAX(CAST(SUBSTR(internal_id, 10) AS INTEGER))
        FROM supplements
        WHERE internal_id LIKE 'SPL-' || ? || '-%'
    `, year).Scan(&maxNum)
    if err != nil {
        return "", err
    }
    next := 1
    if maxNum.Valid {
        next = int(maxNum.Int64) + 1
    }
    return fmt.Sprintf("SPL-%d-%04d", year, next), nil
}
```

### FK Validation

On create/update, validate before INSERT/UPDATE:

1. **contract_id** exists and `deleted_at IS NULL`
2. **client_signer_id** (if provided) exists AND belongs to the contract's client
3. **supplier_signer_id** (if provided) exists AND belongs to the contract's supplier

Signer validation query:
```sql
SELECT COUNT(*) FROM authorized_signers
WHERE id = ? AND deleted_at IS NULL
AND company_id = (SELECT client_id FROM contracts WHERE id = ?)
```

### Status Transition Handler

```go
validTransitions := map[string][]string{
    "draft":    {"approved"},
    "approved": {"draft", "active"},
    "active":   {}, // terminal
}
```

Returns 400 if transition is invalid:
```json
{"error": "cannot transition from 'active' to 'draft'"}
```

### Error Responses

| Scenario | HTTP | Message |
|----------|------|---------|
| Invalid transition | 400 | `"cannot transition from '{from}' to '{to}'"` |
| Contract not found | 400 | `"contract not found"` |
| Client signer not found | 400 | `"client signer not found"` |
| Supplier signer not found | 400 | `"supplier signer not found"` |
| Invalid status value | 400 | `"status must be 'draft', 'approved', or 'active'"` |
| Supplement not found | 404 | `"supplement not found"` |
| DB error | 500 | `"failed to create/update/delete supplement"` |

### Audit Logging

All operations use `h.auditLog()`:
- **create**: newState with all fields
- **update**: prevState + newState (full JSON capture)
- **status transition**: prevState with old status, newState with new status
- **delete**: prevState with supplement data

Failures are silent — audit never breaks the primary operation.

### 2. API Client (`src/lib/api.ts`)

New module:

```ts
const api = (path: string, options: RequestInit = {}) => {
  return fetch(`/api${path}`, {
    credentials: 'include',
    headers: { 'Content-Type': 'application/json', ...options.headers },
    ...options,
  });
};

export const supplementsAPI = {
  list: () => api('/supplements').then(r => r.json()),
  listByContract: (id: number) => api(`/supplements/contract/${id}`).then(r => r.json()),
  create: (data: CreateSupplementRequest) =>
    api('/supplements', { method: 'POST', body: JSON.stringify(data) }).then(r => r.json()),
  update: (id: number, data: UpdateSupplementRequest) =>
    api(`/supplements/${id}`, { method: 'PUT', body: JSON.stringify(data) }).then(r => r.json()),
  transitionStatus: (id: number, status: SupplementStatus) =>
    api(`/supplements/${id}/status`, { method: 'PATCH', body: JSON.stringify({ status }) }).then(r => r.json()),
  delete: (id: number) =>
    api(`/supplements/${id}`, { method: 'DELETE' }).then(r => r.json()),
};
```

### 3. SupplementsPage.tsx — Full Refactor

**Remove:**
- `getSupplements` / `setSupplements` from `@/lib/storage`
- localStorage-based `handleSubmit`, `handleEdit`, `handleDelete`
- `getCurrentUser()` for audit logging (backend handles this)

**Add:**
- `useEffect` loads data via `supplementsAPI.list()`
- `AbortController` for cleanup on unmount
- `useState` for loading and error states
- Status transition buttons in table:
  - `draft` → "Approve" button (blue)
  - `approved` → "Activate" (green) + "Return to Draft" (gray)
  - `active` → no buttons (terminal)
- `supplement_number` displayed in table, `internal_id` as tooltip
- Contract link uses backend contract ID

**Status badge colors:**
- `draft`: gray/secondary
- `approved`: blue/default
- `active`: green/default

### 4. SupplementForm.tsx — Refactor

**Changes:**
- `supplement_number` is a required text input (user enters legal number)
- Contract dropdown loads from `contractsAPI.list()` (not localStorage)
- Signer dropdowns load from `signersAPI.list()`, filtered by contract's client/supplier
- Zod validation schema for all fields
- Submit calls `supplementsAPI.create()` or `supplementsAPI.update()`
- Error display from API responses
- Loading state during submit

### 5. ContractDetailsPage.tsx — Integration

Add supplements section:
- Fetch via `supplementsAPI.listByContract(contractId)`
- Table: supplement_number, status badge, effective_date, actions
- "Add Supplement" button → navigate to SupplementsPage with `?action=create&contractId=X`
- Status transition buttons inline
